# Prototype Spatial Overlay — Technical Specification
## Step 9 — Non-Interactive CAD Overlay, Toyota bZ4X (No AI)

**Roadmap reference:** Phase 1, Deliverable 9
**Status:** `[x]` Spec complete
**Owner:** Engineering Lead (XR)
**Target:** Demo-ready by Month 4 (reviewed by product + engineering leadership)

---

## Objective

Build a **non-interactive** spatial overlay that proves the core spatial premise: a technician can put on a HoloLens 2, look at a Toyota bZ4X, and see accurate CAD geometry of the battery thermal management system overlaid on the physical vehicle — with panels closed.

**What this prototype is NOT:**
- Not connected to OBD-II or any live sensor data
- Not voice-controlled
- Not AI-driven
- Not interactive (no menu navigation)

**What this prototype MUST prove:**
1. CAD geometry renders accurately on the physical vehicle (correct scale, correct position)
2. SLAM anchoring is stable for a 10-minute session without significant drift
3. Layer opacity, color-coding, and LOD switching work as designed
4. A non-technical observer watching a 5-minute demo can understand what they're seeing

## What the Prototype Renders (Fixed Scene, No Dynamic Loading)

For the demo, the following layers are hardcoded in the Unity scene — no runtime loading:

| Layer | Render State | Visual Style |
|---|---|---|
| **Battery TMS coolant loop** | Always visible | Blue emissive material, 80% opacity, animated flow direction |
| **Battery module M4** | Always visible, pulsing | Red emissive, 1Hz pulse cycle (simulates fault state) |
| **Coolant pump assembly** | Always visible | Amber emissive, 70% opacity |
| **HV bus routing** | Always visible | Orange emissive, 60% opacity, dashed line style |
| **Exterior body ghost** | Always visible | White, 12% opacity (provides spatial reference without obscuring overlay) |

## Spatial Anchoring Approach

### Phase 1 Prototype: ArUco Marker-Based Anchoring

**Why ArUco markers (not SLAM alone)?**
Pure SLAM anchoring works well in feature-rich environments but struggles with:
- Shiny, reflective metal surfaces (car body panels)
- Uniform-color surfaces (black engine bays)
- Repeated textures (engine bay grid patterns)

ArUco markers provide absolute anchor points that override SLAM drift, giving us reliable anchoring for the demo without requiring mature SLAM fine-tuning.

**Marker placement (Toyota bZ4X):**

| Marker ID | Placement Location | 3D Position in Vehicle Frame |
|---|---|---|
| `0x01` | Front left corner of hood (taped) | X=0, Y=900, Z=1200 mm from front axle |
| `0x02` | Front right corner of hood | X=0, Y=-900, Z=1200 mm |
| `0x03` | Rear of hood, center | X=800, Y=0, Z=1100 mm |
| `0x04` | Driver door sill (reference for height) | X=1200, Y=850, Z=0 mm |

**Calibration procedure:**
1. Place ArUco markers on vehicle (printed on A4, laminated, taped with painter's tape — non-damaging)
2. Run `MetaDome_Calibration` Unity scene on HoloLens 2
3. Tech scans all 4 markers by looking at them (HoloLens 2 camera detects ArUco via OpenCV)
4. System computes rigid body transform — vehicle coordinate origin registered
5. All CAD geometry now anchored in vehicle-relative space
6. Calibration takes ~90 seconds; stored in Azure Spatial Anchors for the session

**Anchor accuracy target:** CAD geometry within ±15mm of physical component position (validated by measuring rendered component vs physical component with ruler during QA)

## Rendering Specification

### Material Properties

```
Coolant loop material:
  Base color: #2196F3 (Material Blue 500)
  Emissive intensity: 1.5
  Opacity: 0.80
  Rendering mode: Transparent + Additive blend

Module M4 fault material (pulsing):
  Base color: #F44336 (Material Red 500)
  Emissive: animated, 0.5–2.0 over 1s sine curve
  Opacity: 0.85
  Rendering mode: Transparent + Additive blend

HV bus material:
  Base color: #FF6F00 (Amber 900)
  Emissive intensity: 2.0 (intentionally bright — safety visibility)
  Opacity: 0.75
  Line width: 8px world-space (visible from 3m)
  Style: dashed (75% dash, 25% gap)

Ghost body panel material:
  Base color: #FFFFFF
  Emissive intensity: 0.3
  Opacity: 0.12
  Rendering mode: Transparent
```

### LOD Switching

| Distance from Camera | LOD Level | Triangle Budget per Layer |
|---|---|---|
| 0–2m | LOD-0 | 150,000 |
| 2–5m | LOD-1 | 40,000 |
| > 5m | LOD-2 | 8,000 |

LOD transitions use a 0.2m hysteresis band to prevent flickering at distance boundaries.

### Frame Rate Target

- 60fps sustained (HoloLens 2 display is 60Hz)
- GPU budget: < 12ms frame time (leaves headroom for MRTK UI overhead)
- Profile on device; if frame budget exceeded, reduce LOD-0 triangle budget first

## Demo Script (5-Minute Proof-of-Concept)

For leadership review:

**Minute 0–1:** Calibration
- Don headset, run calibration scene, scan 4 ArUco markers on vehicle
- Confirm anchor registered: ghost body overlay appears on vehicle

**Minute 1–3:** Core spatial overlay
- Walk around vehicle — confirm overlay tracks correctly
- Move head close to hood — LOD transition from LOD-1 to LOD-0
- Point out: "Blue = coolant circuit. Red pulsing = Module 4 fault. Orange dashed = HV bus route."
- Demonstrate that the overlay is inside the car — visible through the closed hood

**Minute 3–4:** Drift check
- Remove and re-don headset
- Confirm anchor re-resolves (target: < 10 seconds)
- Walk to opposite side of vehicle — confirm overlay accuracy from new angle

**Minute 4–5:** Q&A + handoff to leadership
